How to verify a signal hash manually

Each signal hash is a SHA-256 digest of the raw captured payload: message text + receive timestamp + source metadata.

echo -n "${raw_payload}|${timestamp}|${source}" | sha256sum

When you retrieve a signal via this lookup, compare the returned hash against your own SHA-256 computation of the raw payload to verify integrity.

How outcomes are judged

Standardized rules applied uniformly across all providers. No manual overrides.

⎆ Entry fill logic

A signal's entry is considered filled when exchange spot price touches or crosses the declared entry within the signal's valid window. Timestamp of first touch is recorded.

⎋ TP / SL precedence

If price hits both a take-profit and stop-loss in the same candle, the outcome is judged by which level was reached first using tick-level data. If indeterminate, the signal is marked ambiguous.

⏱ Time-based expiry

Signals that don't reach any target or stop within 72 hours of entry fill are automatically classified as expired — not win or loss.

⛓ Exchange source of truth

All price verification uses Binance spot market data. If Binance is unavailable for a pair, Kraken is used as fallback. The exchange source is recorded per signal.

✎ Edits and deletions

If a provider edits or deletes a signal after capture, the original payload and hash are preserved. Edit events are logged but do not alter the tracked outcome.

◎ Partial target hits

If TP1 is hit but TP2 is not, the signal resolves as partial_win. PnL is calculated based on the highest target actually reached. Each target hit is individually timestamped.

? Ambiguous signals

Signals that cannot be reliably parsed (missing entry, unclear direction, no targets) are flagged as unparseable and excluded from win/loss statistics.

⚖ Uniform rules

Every provider is subject to identical tracking rules. No provider can opt out, suppress signals, or request manual overrides. The system is automated.
